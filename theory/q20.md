## Q20: setup() and loop() — delay() Problem and Non-Blocking Alternative

### The two core Arduino functions

#### setup()
- Runs **once** when Arduino is powered on or reset
- Used to configure pins, start Serial, initialize sensors
- Think of it as: *"get everything ready before the job starts"*

```cpp
void setup() {
  pinMode(13, OUTPUT);   // set pin mode
  Serial.begin(9600);    // start serial communication
}
```

#### loop()
- Runs **forever** in a continuous cycle after setup() finishes
- All your main logic goes here — reading sensors, checking conditions, controlling outputs
- Think of it as: *"keep doing the job, again and again"*

```cpp
void loop() {
  int val = analogRead(A0);  // read sensor
  Serial.println(val);       // print value
}
```

---

### What happens if you put a long delay() inside loop()?

`delay(ms)` **completely pauses** the Arduino for that many milliseconds. During that pause:
- No sensor can be read
- No button press can be detected
- No Serial command can be received
- Nothing else happens at all

**Example of the problem:**
```cpp
void loop() {
  digitalWrite(13, HIGH);
  delay(5000);              // Arduino is FROZEN for 5 seconds
  digitalWrite(13, LOW);
  delay(5000);              // Frozen again
}
```

If a user sends a Serial command or presses a button during those 5 seconds — Arduino completely **misses it**.

---

### How does this affect sensor reading responsiveness?

In IoT systems, sensors often need to be checked frequently — temperature spikes, motion detected, button pressed. If `delay()` is blocking the loop, the Arduino **cannot respond in real time**.

**Real example:** A fire sensor needs to trigger an alarm immediately. But if your loop has `delay(3000)` for an unrelated LED blink, the fire alert is delayed by up to 3 seconds — dangerous in a real system.

---

### The Non-Blocking Alternative — millis()

`millis()` returns the number of milliseconds since Arduino was turned on. Instead of freezing the program, you **check how much time has passed** and act only when enough time has elapsed.

```cpp
unsigned long lastTime = 0;
int interval = 1000;  // 1 second

void loop() {
  unsigned long currentTime = millis();

  if (currentTime - lastTime >= interval) {
    lastTime = currentTime;
    digitalWrite(13, !digitalRead(13));  // toggle LED
  }

  // Arduino is FREE to do other things here
  int sensorVal = analogRead(A0);  // this runs every loop, not blocked!
  Serial.println(sensorVal);
}
```

The LED still blinks every 1 second — but now the sensor is being read **continuously** without any pause.

---

### Comparison Table

| Feature | delay() | millis() |
|---|---|---|
| Blocks execution | Yes — freezes everything | No — program keeps running |
| Sensor responsiveness | Poor — misses events | Good — reads continuously |
| Button/Serial detection | Can miss inputs | Always responsive |
| IoT suitability | Bad for real-time systems | Recommended for IoT |
| Code complexity | Simple | Slightly more code |

---

### Summary
- `setup()` runs once — for initialization
- `loop()` runs forever — for main logic
- `delay()` freezes Arduino — bad for IoT responsiveness
- `millis()` is the non-blocking alternative — lets Arduino multitask
